# Об'єднання проєктів — звіт про зміни

**Мета:** зберегти 100% функціоналу робочого проєкту `dryleaf`, але повністю
замінити зовнішній вигляд клієнтських екранів на новий дизайн зі статичного
макета `sitemenudryleaf-main`.

---

## 1. Що це були за проєкти (результат аналізу архітектури)

| | `dryleaf` | `sitemenudryleaf-main` |
|---|---|---|
| Тип | Повноцінний **Next.js 14** застосунок (App Router) | **Статичний** макет: `index.html` + `styles.css` + `script.js` + `data.js` |
| Backend / API | ✅ Next.js Route Handlers, Server Actions | ❌ немає |
| БД | ✅ Prisma + PostgreSQL | ❌ дані захардкоджені в `data.js` |
| Авторизація | ✅ NextAuth (Credentials, JWT) | ❌ |
| Корзина / checkout | ✅ Zustand + Stripe | ❌ (тільки телефон для замовлення) |
| Адмінка | ✅ дашборд, меню, замовлення, брони, столики | ❌ |
| Дизайн-система | ✅ Tailwind (base/sage/gold, Cormorant+Inter) | ✅ **ті самі** токени та шрифти |

**Ключовий висновок:** обидва проєкти побудовані на **одній дизайн-системі**.
Статичний макет — це уточнена візуальна специфікація трьох клієнтських екранів
(вибір закладу → меню → модалка страви). Тому «об'єднання» = взяти робочий
`dryleaf` за основу і привести його клієнтський UI у точну відповідність макету,
**нічого не втративши** з функціоналу. Дублювання проєктів чи заміни не робилось.

---

## 2. Змінені файли

### Клієнтський UI (переписано під новий дизайн)
- **`src/components/client/HeroSection.tsx`** — головний екран вибору закладу:
  - два логотипи в шапці (`/logo-left.png`, `/logo-right.png`);
  - eyebrow «Меню закладу», заголовок `Dry Leaf`, розділювач;
  - підзаголовок «Оберіть локацію, щоб переглянути меню»;
  - картки закладів: радіус 18px, фото 150px, тег закладу, CTA «Переглянути меню»;
  - футер із двома рядками («Доставляємо…» / «Фото в меню не відповідають дійсності»);
  - декоративні лінії на 28% / 30%. Анімації входу (framer-motion) збережено.
- **`src/components/client/DishCard.tsx`** — картка страви нового дизайну:
  - фото зі співвідношенням 72%, назва (serif 15px, 2 рядки), ціна ₴ золотом, порція;
  - **збережено** швидке додавання в корзину (кнопка «+», лічильник) і відкриття модалки.
- **`src/components/client/DishModal.tsx`** — bottom-sheet страви:
  - верхній eyebrow-категорії, заголовок 30px, опис, блок «ціна + порція»;
  - **збережено** додавання в корзину, лічильник кількості, коментар, алергени, склад.
- **`src/app/v/[venueSlug]/page.tsx`** — сторінка меню:
  - стиль заголовків категорій під макет (serif 22px + градієнтна лінія);
  - передача в картку `categoryName` та нового поля `size`.

### Нові файли
- **`src/components/client/DishImage.tsx`** — зображення страви/закладу зі
  «м'яким» SVG-фолбеком (як у макеті). Використовує звичайний `<img>` замість
  `next/image`, щоб не залежати від `remotePatterns` для довільних хостів фото.
- **`src/lib/placeholder.ts`** — генератор елегантного SVG-плейсхолдера з логотипом-
  листком і назвою страви (портовано з `sitemenudryleaf-main/script.js`).
- **`public/logo-left.png`, `public/logo-right.png`, `public/dryleaf-hero.png`** —
  ассети з макета.
- **`prisma/menu-data.json`** — повне меню обох закладів (2 заклади, 23 категорії,
  **454 страви**), згенероване з `sitemenudryleaf-main/data.js`.
- **`.gitignore`** — стандартний для Next.js.
- **`CHANGES.md`** — цей звіт.

### Конфігурація
- **`next.config.mjs`** — до `images.remotePatterns` додано wildcard-хости
  (`**`), бо реальні фото меню розміщені на десятках зовнішніх доменів; тепер
  вони коректно вантажаться і через `next/image` (кошик, адмінка, підтвердження).

### БД / бекенд (мінімальні, зворотно-сумісні правки)
- **`prisma/schema.prisma`** — додано **два опційні (nullable) поля**:
  - `Venue.tagline String?` — короткий підпис на картці закладу;
  - `Dish.size String?` — порція (напр. «400г»), яку показує новий дизайн.
- **`src/server/menu.ts`** — `getVenues()` тепер віддає `tagline` (для карток hero).
  `getVenueWithMenu()` вже включає повний `Dish`, тож `size` підтягується автоматично.
- **`src/server/admin.ts`** — у Zod-схему `DishSchema` додано `size` (опційно).
- **`src/components/admin/MenuEditor.tsx`** — у формі редагування страви з'явилось
  поле «Порція» (запис/читання `size`).
- **`prisma/seed.ts`** — повністю переписано: заповнює БД **реальним меню** з
  `menu-data.json` (2 заклади з `tagline`, 23 категорії з групами «Кухня»/«Бар»,
  454 страви з назвами, описами, порціями, цінами та фото). Ціни з `data.js`
  (у гривнях) переведено в копійки (`priceCents = грн × 100`). Плюс демо-
  замовлення та бронювання для дашборда адмінки.

### Довідкові матеріали (не входять у збірку)
- **`design-reference/new-design-static/`** — оригінальний статичний макет.
- **`design-reference/dryleaf-preview.html`** — вихідний HTML-прототип із кореня `dryleaf`.

### Видалено
- Порожня директорія-артефакт `{prisma,src` (наслідок невдалого `mkdir` без
  розкриття фігурних дужок) — прибрана.

---

## 3. Додані залежності

**Жодних.** Новий дизайн реалізовано на вже наявному стеку
(Tailwind, framer-motion, lucide-react, next/image). `package.json` не змінювався.

---

## 4. Вирішені конфлікти

- **Конфлікт «дизайн vs функціонал» у модалці страви.** Макет показує в модалці
  блок «контакти/доставка» замість кнопки замовлення (бо статичний сайт — тільки
  вітрина). Робочий проєкт має корзину. Рішення: залишено вигляд нового дизайну
  (eyebrow-категорія, типографіка, ціна+порція), але **збережено** робочу кнопку
  «Додати до кошика», лічильник і коментар. Нічого функціонального не втрачено.
- **Різна поведінка зображень.** Макет використовує `<img>` з `onerror` → SVG-
  плейсхолдер; проєкт — `next/image` (потребує білого списку доменів). Щоб
  відтворити поведінку макета й уникнути 404 на довільних хостах фото, для страв
  введено `DishImage` (звичайний `<img>` + SVG-фолбек). `next/image` лишився там,
  де домени вже дозволені (`next.config.mjs`).
- **Відсутнє поле «порція».** Дизайн показує розмір порції, а моделі `Dish` його
  бракувало. Додано як **опційне nullable** поле → існуючі БД лишаються сумісними
  (`prisma db push` додасть колонку без втрати даних).
- **Артефакт копіювання** `{prisma,src` — прибрано.
- **Маршрути/API не змінювались** — всі роути (`/`, `/v/[venueSlug]`, `/v/.../checkout`,
  `/v/.../order/[orderId]`, `/v/.../reserve`, `/t/[tableCode]`, `/admin/*`,
  `/api/auth/*`, `/api/checkout`, `/api/stripe/webhook`) працюють як раніше.

---

## 5. Сумісність БД

Зміни адитивні й зворотно-сумісні: два **nullable** стовпці. Для наявної бази
достатньо `npm run db:push` (або нова міграція) — дані не втрачаються, старі рядки
матимуть `NULL` у нових полях, а UI коректно їх приховає.

---

## 6. Перевірки якості

- ✅ Строгий `tsc` по всьому написаному UI-коду (HeroSection, DishCard, DishModal,
  DishImage, placeholder, cart, money, utils) — без помилок.
- ✅ Синтаксичний розбір (TypeScript API) файлів, залежних від Prisma
  (сторінка меню, MenuEditor, server/*, seed) — без помилок.
- ✅ Перевірка балансу дужок/JSX у всіх змінених файлах.
- ℹ️ Повний `next build` / `prisma generate` у цьому середовищі не виконувався:
  рушій Prisma тягнеться з `binaries.prisma.sh`, недоступного з пісочниці.
  На машині з доступом до мережі типи Prisma-шару підтвердяться під час
  `npm install` (крок `postinstall: prisma generate`).

---

## 7. Запуск об'єднаного проєкту

```bash
# 1. Залежності (виконає prisma generate у postinstall)
npm install

# 2. Змінні оточення
cp .env.example .env
#   і заповни:
#   - DATABASE_URL           (PostgreSQL: локальний / Supabase / Neon)
#   - NEXTAUTH_SECRET        (openssl rand -base64 32)
#   - NEXTAUTH_URL           (http://localhost:3000)
#   - STRIPE_SECRET_KEY, NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY, STRIPE_WEBHOOK_SECRET
#   - SEED_ADMIN_EMAIL, SEED_ADMIN_PASSWORD

# 3. Схема БД + демо-дані
npm run db:push      # створює/оновлює таблиці (додасть tagline та size)
npm run db:seed      # заповнює 2 заклади, меню, столики, тест-замовлення, адміна

# 4. Dev-сервер
npm run dev          # http://localhost:3000

# (опційно) Stripe-вебхук у окремому терміналі:
# stripe listen --forward-to localhost:3000/api/stripe/webhook

# Продакшн:
npm run build && npm start
```

**Точки входу:**
- `/` — новий екран вибору закладу;
- `/v/dry-leaf` та `/v/citadel` — меню в новому дизайні;
- `/admin/login` — вхід в адмінку (email/пароль із сідера).
